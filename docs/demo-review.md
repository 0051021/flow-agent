# DeskClaw Enterprise Demo 评审意见

> 评审对象：https://deskclaw-enterprise.surge.sh
> 评审日期：2026-04-02
> 评审视角：AI 产品经理（业务翻译 + 架构设计）

---

## 一、总体评价

这个 demo 是一份**功能全景图**，而不是一份**产品设计方案**。它展示了"我们可能需要哪些模块"，但没有回答"用户怎么用、数据怎么流转、核心体验是什么"。

**核心问题：有功能，没逻辑。**

产品经理的交付物应该是：用户故事 → 信息架构 → 核心流程 → 页面原型 → 交互细节。这个 demo 跳过了前三步，直接画了页面。

---

## 二、逐模块评审

### 2.1 首页（工作台）

**现状**：
- 录屏 + 对话共创（核心 CTA）
- AI 周报卡片
- 热门场景卡片（6 个）
- 统计数字（33 预设场景、6 任务类型、3 执行架构）

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | "录屏 + 对话共创"作为首页核心入口不合理 | 高 | 这是一个技术实现难度极高的功能（实时视频理解 + 对话 + 流程提取），不应作为 MVP 的主入口。它给用户的预期是"这是产品最重要的功能"，但实际上短期内无法交付。 |
| 2 | AI 周报没有数据来源 | 中 | 周报内容是静态 mock，但产品设计中没有说明：数据从哪来？多久更新？谁能看？格式是否可定制？ |
| 3 | 热门场景的排序逻辑不明 | 低 | 6 个场景的排列依据是什么？按使用频率？按推荐度？按行业？没有说明。 |

**建议**：
- 首页核心入口应该是**自定义场景**（对话式业务翻译），这是产品的核心价值
- "录屏 + 对话"降级为高级功能入口，放在场景创建流程的可选步骤中
- AI 周报改为"待处理事项"面板（待确认任务、异常告警），更有行动指导性

---

### 2.2 场景库

**现状**：
- 33 个预设场景，按 6 大类分组
- 每个场景有详情页：任务类型、架构、复杂度、预估交付、典型输入输出、所需 Skill

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | 33 个场景全是静态模板，没有一个能跑通 | 高 | 每个场景详情页的"开始使用"按钮点击后没有后续流程。这意味着产品经理在广度上花了力气，但没有打通任何一个场景的完整链路。**一个能跑通的场景 > 33 个静态卡片。** |
| 2 | Workflow 和 Agentic 场景的详情页完全一样 | 高 | "财务报销"（Workflow）和"账号运营"（Agentic）的详情页结构完全相同，都是"典型输入/输出/所需 Skill"。但这两种任务类型的确认方式、执行方式、监控方式完全不同。**这是最大的设计缺失。** |
| 3 | "预估交付 2-3 周"缺乏依据 | 中 | 交付周期的估算没有说明方法论。是基于什么经验？哪些因素影响？用户看到这个数字会怎么理解？ |
| 4 | 场景分类缺少"混合型" | 中 | 会议中明确讨论了一个业务可能同时包含 Workflow 和 Agentic 节点，但场景库的分类是互斥的。 |
| 5 | 缺少场景的"适配度评估" | 中 | 用户怎么知道自己的业务适不适合用这个平台？会议中强调"选错场景会使工作无价值"，但 demo 没有任何引导。 |

**建议**：
- 砍到 6 个场景（每种类型 1 个代表），但每个都要有完整的交互流程
- Workflow 场景详情页展示流程图预览 + "开始翻译"入口
- Agentic 场景详情页展示任务配置预览（目标/Skill/评估器）+ "开始配置"入口
- 增加"场景适配度自检"功能：用户回答 3-5 个问题，系统判断适配度

---

### 2.3 自定义场景

**现状**：
- 一个输入框 + "AI 分析"按钮
- 左侧有 7 步流程侧边栏（场景选择 → 场景评估 → 信息采集 → AI Plan → 共创确认 → 通知配置 → 输出）

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | 7 步流程只有第 1 步有界面 | 高 | 侧边栏列了 7 个步骤，但只有"场景选择"有内容，其余 6 步都是空白。这是最典型的"功能堆砌"——列了名字但没有设计。 |
| 2 | 业务翻译的核心交互缺失 | 高 | 会议反复强调"不要指望业务直接给 AI 讲明白"，需要引导式输入。但这里就是一个空白输入框，等于把最难的部分完全交给了用户。 |
| 3 | "AI 分析"按钮的输出是什么？没有定义 | 高 | 用户输入描述后点"AI 分析"，然后呢？生成什么？展示在哪里？用户怎么确认？怎么修改？完全没有设计。 |
| 4 | 7 步流程的逻辑关系不清晰 | 中 | "场景选择"和"场景评估"是什么关系？"AI Plan"和"共创确认"的区别是什么？步骤之间的数据流转没有定义。 |

**建议**：
- 这 7 步应该有明确的输入/输出定义：

| 步骤 | 输入 | 输出 | 交互方式 |
|------|------|------|---------|
| 场景描述 | 用户自然语言 | 结构化场景摘要 | 对话式引导 |
| 类型判断 | 场景摘要 | Workflow / Agentic / 混合 | AI 判断 + 用户确认 |
| 信息采集 | 类型判断结果 | 补充的业务细节 | 按类型不同的问答模板 |
| AI Plan | 完整业务信息 | 流程图草稿 / 任务配置草稿 | AI 生成 |
| 共创确认 | 草稿 | 确认后的最终方案 | 逐节点确认（Workflow）/ 逐模块确认（Agentic）|
| 通知配置 | 确认方案 | 通知规则 | 表单配置 |
| 输出交付 | 全部 | 可执行的配置文件 | 预览 + 导出 |

### 2.4 Agent 团队

**现状**：
- 7 个 Agent 卡片，展示成功率、任务数、均耗时
- AI 团队诊断卡片
- 版本号、活跃时间

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | Agent 的创建/配置流程缺失 | 高 | 页面展示了已有的 Agent，但用户怎么创建一个新 Agent？从场景到 Agent 的转化过程是什么？ |
| 2 | Agent 和场景的关系不明确 | 中 | "报关 Agent"对应"进出口报关"场景，但这个对应关系在哪里建立？一个 Agent 能处理多个场景吗？ |
| 3 | "AI 团队诊断"的触发和频率未定义 | 低 | 是实时的？每天一次？用户主动触发？ |

**建议**：
- Agent 团队页面应该是管控后台的模块，不是业务翻译工具的模块
- 需要定义清楚：场景（FlowAgent 的输出）→ Agent（运行时实体）的转化流程

---

### 2.5 任务监控

**现状**：
- 任务列表表格：ID、Agent、当前节点、进度、状态、耗时
- AI 值班主管卡片
- 状态筛选（全部/执行中/待确认/已完成/异常/排队中）

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | "待确认"任务的确认交互缺失 | 高 | 表格里有 3 个"待确认"任务，但点击后应该展示什么？用户怎么确认？确认后任务怎么继续？这是人机协作的核心交互，完全没有设计。 |
| 2 | 任务详情页缺失 | 高 | 点击任何一个任务，应该能看到完整的执行链路（哪个节点在执行、每个节点的输入输出、执行日志）。会议强调"过程不可黑盒化"，但 demo 里任务就是一行表格。 |
| 3 | 异常处理流程缺失 | 中 | 2 个异常任务，但异常后怎么处理？重试？人工接管？跳过？回退？ |

**建议**：
- 任务详情页是管控后台最核心的页面，需要优先设计
- 至少包含：执行时间线、节点级日志、人工确认入口、异常处理操作

---

### 2.6 数据驾驶舱

**现状**：
- 4 个统计卡片（活跃 Agent、月任务量、自动化覆盖率、月 Token 消耗）
- 本周任务趋势图
- Agent 排行

**问题**：

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | 指标缺乏业务意义 | 中 | "月 Token 消耗 ¥2,340"对业务方没有意义。应该展示"节省人力 X 人/天"、"平均处理时间缩短 X%"等业务指标。 |
| 2 | 没有下钻能力 | 中 | 看到"报关 Agent 成功率 94.2%"后，用户想知道为什么下降了，但没有下钻到具体任务/节点的能力。 |

**建议**：
- 驾驶舱是 Phase 2 的功能，当前不需要投入
- 如果要做，核心指标应该是：业务目标达成率、人工介入率、异常率趋势

### 2.7 审计中心

**现状**：
- 时间线形式的操作日志
- AI 异常检测标记

**评价**：
- 结构合理，但属于 Phase 3 的功能
- 当前阶段不需要投入

---

### 2.8 知识中心

**现状**：
- 10 篇知识文档，按类型分组（术语词典/业务规范/历史案例/SOP 文档）
- 关联场景数、被引用次数

**评价**：
- 和 FlowAgent 现有的 KnowledgePanel 功能重叠
- 知识中心应该是业务翻译工具的一部分（辅助翻译），而不是独立模块
- 当前的 mock 数据结构合理，可以复用

---

### 2.9 技能市场

**现状**：
- 15 个 Skill，分三层（通用/行业/定制）
- 每个 Skill 有：输入/输出/评估指标/调用次数/平均耗时/可用状态

**评价**：
- **这是 demo 中设计最好的模块**
- Skill 的数据结构（name/input/output/evaluator/status）和会议讨论一致
- 三层架构（通用 → 行业 → 定制）清晰
- 但缺少：Skill 的创建/编辑流程、Skill 与场景的绑定关系、Skill 的版本管理

**建议**：
- 技能市场应该作为独立的基础设施服务
- FlowAgent 在配置 Agentic 任务时，通过 API 调用技能市场来选择 Skill

### 2.10 系统设置

**现状**：
- 角色模板（6 种角色）
- 权限矩阵（成员 × 模块的权限表）
- AI 权限分析

**评价**：
- RBAC 权限模型设计合理
- 但属于 Phase 3 的功能，当前不需要投入

## 三、缺失的关键设计

以下是会议中明确讨论但 demo 中完全没有体现的内容：

| # | 缺失内容 | 重要程度 | 会议中的讨论 |
|---|---------|---------|------------|
| 1 | **Workflow vs Agentic 的差异化交互** | 极高 | 会议明确了两种架构（Workflow + Agent Tech），但 demo 中所有场景的交互完全一样 |
| 2 | **业务翻译的引导式输入** | 极高 | "不要指望业务直接给 AI 讲明白"，需要模块化、程序化的引导方式 |
| 3 | **人工确认/干预的交互设计** | 高 | "人机协作过程要避免黑盒化"，但 demo 中没有任何确认/干预的交互 |
| 4 | **通知集成的交互流程** | 高 | "通过钉钉或飞书通知，用户点击链接可查看任务全过程"，但 demo 中没有通知相关的设计 |
| 5 | **节点出错的处理方式** | 中 | "回溯、报警或忽略"，但 demo 中异常任务没有处理流程 |
| 6 | **评估器的配置界面** | 中 | 会议讨论了评估器对 Agentic 任务的重要性，但 demo 中没有评估器的配置入口 |
| 7 | **混合任务类型的处理** | 中 | 一个业务可能同时包含 Workflow 和 Agentic 节点 |

## 四、优先级建议

### 立即要做（P0）

1. **打通 1 个 Workflow 场景的完整链路**：从输入 → 翻译 → 确认 → 输出
2. **打通 1 个 Agentic 场景的完整链路**：从输入 → 类型判断 → 配置 → 确认 → 输出
3. **设计 Workflow 和 Agentic 的差异化右侧面板**

### 短期要做（P1）

4. 任务监控的核心交互（任务详情页、人工确认流程）
5. 技能市场的 Skill 选择集成（在 Agentic 配置中调用）
6. 通知配置的基础流程

### 可以延后（P2）

7. 数据驾驶舱
8. 审计中心
9. 权限管理
10. 录屏 + 对话共创

## 五、对产品经理的建议

### 交付物应该是什么

产品经理的核心交付物不是一个"看起来完整的 demo"，而是：

1. **用户故事**：谁在什么场景下用这个功能，要达成什么目标
2. **核心流程图**：用户从进入到完成的每一步，包括正常流程和异常流程
3. **信息架构**：页面之间的层级关系和导航逻辑
4. **交互规格**：每个关键操作的输入、输出、状态变化、错误处理
5. **数据流定义**：每个模块的数据从哪来、到哪去、格式是什么

### 当前 demo 的定位

这个 demo 可以作为**产品愿景展示**（给老板/客户看全景），但不能作为**开发依据**（给工程师看怎么做）。

要变成开发依据，需要补充：
- 每个模块的 PRD（产品需求文档）
- 核心流程的交互稿（不是页面截图，是带状态的流程）
- 数据模型定义（每个实体的字段、关系、状态机）
- API 接口定义（前后端约定）

### 建议的工作方式

> "先把一个场景做到 100 分，再去做第二个场景。不要 33 个场景都做到 10 分。"

具体来说：
1. 选一个 Workflow 场景（如财务报销）和一个 Agentic 场景（如账号运营）
2. 为这两个场景分别写完整的 PRD
3. 画出完整的交互流程（包括异常处理）
4. 和技术方一起评审可行性
5. 然后再扩展到其他场景
