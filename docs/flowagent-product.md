# FlowAgent 产品说明

> 最后更新：2026-04-03（v2）

## 一、FlowAgent 是什么

FlowAgent 是一个 **AI 驱动的业务翻译平台**。

它解决的核心问题是：**业务人员有想法但说不清楚，技术人员能做但不知道做什么。**

传统做法是业务方写需求文档 → 产品经理翻译 → 技术方评估 → 反复沟通。这个过程通常需要 1-2 周，而且信息在每次传递中都会失真。

FlowAgent 让业务方用自然语言描述业务场景，AI 自动翻译成结构化的可执行方案，技术方在同一个平台上直接评审。**从"描述"到"方案确认"，可以在一次对话中完成。**

---

## 二、设计原则

> 以下原则来源于学术研究验证与产品实践反馈，是 FlowAgent 全部产品决策的基础。

### 2.1 增强而非替代

AI 的角色是**增强人的能力**，而非替代人的判断。

- **产品层面**：FlowAgent 帮业务方翻译需求，但人来确认和决策
- **方案层面**：FlowAgent 生成的业务方案本身也应遵循此原则——AI 负责数据处理、内容初稿、定时执行等确定性步骤；人负责质量判断、方向决策、审美把控等需要主观判断的步骤

研究表明（CMU/Stanford 2025）：AI 增强（嵌入现有工作流辅助某一步）效率提升 24.3%，AI 自动化（让 AI 主导全流程）效率反而下降 17.7%。纯自动化方案下，AI 产出的工作质量比人类低 32-50%，主要因为数据捏造和工具滥用。

### 2.2 方案必须体现人机分工

FlowAgent 生成的每一个方案，都应该清晰标注哪些步骤由 Agent 执行、哪些步骤需要人工介入。不允许生成"全自动化"方案——这不是产品能力的妥协，而是基于实证的设计原则。

AI 应在以下类型的步骤后主动建议人工确认节点：

| 步骤类型 | 人工确认理由 |
|----------|-------------|
| 数据采集/处理后 | Agent 可能捏造看似合理但实际错误的数据 |
| 策略/决策类步骤后 | 方向性判断需要业务经验，AI 缺乏上下文 |
| 内容创作后 | 品牌调性和审美需要人把控 |
| 对外发布/不可逆操作前 | 不可回滚的操作必须有最终确认 |
| 长周期任务的阶段节点 | 防止执行方向偏移逐步累积 |

### 2.3 以工作流步骤为协作单元

人机协作的最优颗粒度是"工作流步骤级别"——不是整个任务一次性交给 Agent，也不是每个鼠标点击都要人确认。FlowAgent 的节点级批注、节点级确认、节点级人工介入都遵循这个原则。

## 三、核心能力

### 3.1 AI 业务翻译

业务方在输入框中描述场景（比如"我想自动化进出口报关流程"），AI 会：

1. **需求澄清**（如有必要）— 当输入信息不足时，先追问 2-3 个关键问题，确认需求范围和约束
2. **判断任务类型** — 自动识别这是 Workflow、Agentic 还是混合型任务
3. **生成结构化方案** — 生成包含人机分工的方案（自动标注人工确认节点）
4. **逐项确认** — 对不确定的部分逐个询问业务方，确保方案准确

### 3.2 三种任务类型

| 类型 | 本质 | 适用场景 | AI 输出 | 确认方式 | 示例 |
|------|------|---------|---------|---------|------|
| **Workflow** | 类似 COT（思维链），追求每步准确 | 有明确步骤顺序的确定性流程 | 可视化流程图（节点 + 连线） | 逐节点确认"这步对不对" | 进出口报关、财务报销 |
| **Agentic** | 类似 ReAct，追求目标达成 | 目标导向、需要 AI 自主规划的任务 | 任务配置（目标/技能/约束/评估器） | 确认目标/约束/评估器"方向对不对" | 小红书运营、竞品分析 |
| **混合型** | Workflow + Agentic 节点嵌套 | 部分步骤确定性、部分需 AI 自主规划 | 流程图中部分节点为 Agentic 节点 | Workflow 节点逐个确认 + Agentic 节点确认目标和约束 | 大部分真实业务场景 |

**Workflow vs Agentic 的核心区别**：

| | Workflow (COT) | Agentic (ReAct) |
|---|---|---|
| 核心追求 | **准确性** — 每步都对 | **达成性** — 目标完成 |
| 失败模式 | 某一步错了 → 后面全错 | 可能走弯路但最终可能达到 |
| 人工介入时机 | 节点级 — 某步出错就暂停 | 阶段级 — 关键里程碑暂停 |
| 执行确定性 | 高 — 每次跑结果一样 | 低 — 每次跑路径可能不同 |

**混合型方案的执行模型**：Agentic 节点不是独立的外部 Agent，而是共享整个任务执行上下文的 Agent——它能看到前面 Workflow 步骤的所有输出，也知道后续步骤需要什么格式的输入。它是执行链路里的一段"自由发挥区间"，前后有确定性 Workflow 步骤约束其输入输出。

AI 会自动判断类型，业务方不需要理解技术概念。超出 Workflow/Agentic/混合型三类之外的场景不在产品覆盖范围内。

### 3.3 双角色协作

平台区分两种角色：

| 角色 | 做什么 | 看到什么 |
|------|--------|----------|
| **业务方** | 描述需求、编辑方案、提交评审 | 流程图/配置编辑器、知识中心（业务文档） |
| **技术方** | 评审可行性、添加批注、通过/打回 | 方案视图、批注面板、知识中心（业务文档 + 技术参考） |

两种角色通过不同入口进入，在同一个方案上协作，避免信息在传递中失真。

### 3.4 管控后台

方案确认后，可以提交到管控后台。管控后台提供：

- **运营仪表盘** — 一眼看到有多少 Agent 在运行、任务成功率、待处理事项
- **任务监控** — 每个任务的执行进度、当前节点、耗时
- **人工介入** — 需要人工确认的节点会暂停等待，异常任务可以重试/接管/跳过

## 四、FlowAgent 的输出定位

FlowAgent 的输出不是"点一下就能跑的成品"，而是**结构化的、业务和技术已对齐的方案蓝图**。

技术人员拿到蓝图后：

| 方案部分 | 当前阶段 | 未来（Skill 标准化后） |
|----------|---------|---------------------|
| Workflow 部分 | 参考蓝图在 Dify/LangGraph 手动配置 | FlowAgent 输出 Dify 兼容 JSON，技术方审核后导入 |
| Agentic 部分 | 参考蓝图配置 Agent 运行参数 | 仍需技术方配置，因为 Agent planning 的标准化尚未解决 |
| 人工确认节点 | 在执行平台配置回调/通知 | 统一编排层自动处理 |

**关键变量是 Skill 的标准化程度**。如果 Skill 只有名字（"关税查询"），必须手动搭建。如果 Skill 有完整的 API 定义（输入/输出 schema、调用方式、错误处理），自动生成就成为可能。Workflow 的自动化程度天然高于 Agentic，因为确定性步骤更容易映射到标准化 Skill。

## 五、知识体系与反馈闭环

### 5.1 双层知识架构

| 层级 | 内容 | 来源 | 更新方式 |
|------|------|------|---------|
| **静态知识** | 行业模板、最佳实践、API 文档、SOP | 人工录入 + SOP 文档自动提取 | 低频，人工审核后入库 |
| **动态知识** | 执行日志、技术方评审批注、异常归因标注 | 系统自动采集 + 技术方标注 | 自动积累 |

### 5.2 知识沉淀机制

知识沉淀的来源不是业务方填表，而是系统和技术方自然产生的数据：

| 来源 | 能判断什么 | 沉淀什么 |
|------|-----------|---------|
| **系统自动采集** | 节点执行成功/失败、耗时、错误日志 | 执行指标数据（Phase 2 对接 Agent 运行时后） |
| **技术方** | 失败归因（搭建质量 vs 链路设计 vs 环境问题） | 异常归因标注、评审批注、最佳实践 |
| **业务方** | 最终业务结果是否达预期 | 通过下一次翻译对话自然反馈（"上次方案效果不好，想调整..."） |

### 5.3 AI 翻译参考历史方案

AI 翻译新场景时，参考的是**技术方已确认通过的方案**（状态为 confirmed），而不是所有历史数据。沉淀格式为"方案骨架 + 关键决策点"：

```
场景: 进出口报关自动化
类型: workflow
骨架: [单证采集, 合规校验, 申报提交, 状态跟踪, 异常处理]
关键决策:
  - 合规校验: 用规则引擎而非AI判断，因为合规标准是确定性的
  - 异常处理: 设了人工确认节点，因为海关驳回原因复杂需要人判断
确认人: tech_reviewer_id
确认时间: 2026-04-01
```

### 5.4 SOP 文档自动提取（规划中）

支持上传现有 SOP 文档（Word/PDF/飞书文档），AI 自动提取步骤结构生成 Workflow 草稿，用户在编辑器中确认和修改。比从零口述效率更高。

## 六、引导式输入

### 6.1 问题

会议记录明确指出"不要指望业务方直接讲明白"。当前首页是一个空输入框，缺少引导。

### 6.2 设计思路

引导式输入解决的是**翻译前的信息补全**（确认题目），不同于生成后的逐项确认（确认答案）。

分两种情况处理：

**情况 A：需求清晰（快思考）**
- 用户输入"每月自动汇总考勤数据发给 HR"
- AI 判断信息充分，直接进入分类 → 生成
- 类似 Cursor 简单任务直接进 Agent 模式

**情况 B：需求模糊（慢思考）**
- 用户输入"我想提升部门效率"
- AI 先追问 2-3 个关键问题（"哪个环节最耗时？现在怎么做的？期望达到什么效果？"）
- 收集到足够信息后再分类 → 生成
- 类似 Cursor 复杂任务先进 Plan 模式

AI 在分类阶段同时判断**需求清晰度**：清晰则直接生成，模糊则进入追问模式。追问精简到 2-3 个问题，避免沟通成本抵消效率收益。

**辅助引导**：输入框下方显示结构化思维提示——"试试描述清楚：你要解决什么问题？现在怎么做的？期望结果是什么？"

| 流程图编辑器 | ✅ 可用 | 拖拽、添加/删除节点、编辑节点详情、画布缩放 |

| 双角色体系 | ✅ 可用 | 业务方/技术方独立入口，编辑器内角色标签 + 切换 |
| 技术方评审 | ✅ 可用 | 4 个 Mock 场景直接加载，批注/回复、评审通过/打回 |
| 批注系统 | ✅ 可用 | 按节点批注（Workflow）/ 全局批注（Agentic），回复、标记已解决 |
| 知识面板 | ✅ 可用 | 业务文档 + 技术参考（技术方双 Tab） |
| 提交至管控后台 | ✅ 可用 | 生成配置 JSON 摘要，可复制 |
| 管控后台仪表盘 | ✅ 可用 | 统计卡片、待处理事项、最近任务、Agent 概览 |
| Agent 团队管理 | ✅ 可用 | Agent 卡片列表（状态/成功率/任务数） |
| 任务监控 | ✅ 可用 | 筛选表格 + 任务详情页（时间线 + 人工确认/异常处理） |

### 当前限制

| 限制 | 说明 |
|------|------|
| 无真实后端 | 管控后台的数据全部是 Mock，没有真实的 Agent 运行时 |
| 无用户系统 | 角色通过 URL 区分，没有登录/注册 |
| 无数据持久化 | 数据存在浏览器 localStorage，刷新可能丢失 |
| 技术方评审为 Mock | 4 个评审场景是预设数据，不是业务方真实提交的 |
| 无通知集成 | 没有钉钉/飞书通知能力 |
| 方案未体现人机分工 | AI 生成方案时尚未主动插入人工确认节点 |
| 无混合型任务支持 | 不支持 Workflow 中嵌套 Agentic 节点 |
| 无引导式输入 | 首页为空输入框，缺少需求澄清环节 |
| 无历史方案参考 | AI 翻译时不参考已确认方案 |

## 八、用户使用路径

### 业务方路径

```
1. 打开首页 (/)
2. 在输入框描述业务场景（或点击示例卡片，或上传 SOP 文档）
3. [如需求模糊] AI 追问 2-3 个关键问题，确认需求
4. AI 自动判断类型（Workflow / Agentic / 混合型），生成包含人机分工的方案草稿
5. 逐项确认 AI 不确定的部分
6. 在编辑器中查看和修改方案
7. 满意后点击「提交至管控后台」
8. 查看生成的配置 JSON 摘要
```

### 技术方路径

```
1. 打开技术方入口 (/tech)
2. 查看待评审列表（统计卡片 + 评审项）
3. 点击某个评审项，进入编辑器
4. 查看业务方提交的方案（流程图或 Agentic 配置）
5. 打开批注面板，添加技术评审意见
6. 打开知识面板，查看业务文档和技术参考
7. 点击「评审通过」或「打回修改」
8. [确认后] 参考方案蓝图在 Dify/LangGraph 中搭建
```

## 九、产品价值

### 对业务方

- **降低表达门槛** — 不需要写 PRD，用自然语言描述即可；需求模糊时 AI 主动引导
- **即时可视化** — 说完就能看到流程图/配置方案，不用等产品经理翻译
- **减少沟通成本** — 在同一个平台上和技术方协作，不用反复开会
- **方案可信** — 每个方案都明确标注了人机分工，不是不切实际的"全自动化"

### 对技术方

- **结构化输入** — 拿到的不是模糊需求，而是有明确步骤、输入输出、约束条件的结构化方案蓝图
- **评审有据可依** — 每个节点的执行方式、预估耗时、错误处理、人机分工都已定义
- **批注留痕** — 评审意见直接挂在方案上，不会丢失
- **搭建有参考** — 已确认方案沉淀为模板，后续类似场景可复用

### 对组织

- **缩短需求确认周期** — 从"周级"缩短到"小时级"
- **减少信息失真** — 业务方的原始描述和 AI 翻译结果在同一个界面
- **积累组织知识** — 技术方确认的方案自动沉淀为知识库，AI 翻译质量持续提升
- **人机分工合理** — 方案天然内置人工确认节点，避免"全自动化后出错再返工"

## 十、后续优化计划

### Phase 1：补齐核心体验（1-2 周）

| 优先级 | 事项 | 目标 |
|--------|------|------|
| P0 | **用户系统** | 基础登录/注册，角色绑定到账号而非 URL |
| P0 | **数据持久化** | 方案保存到服务端数据库，支持历史版本 |
| P0 | **业务方提交 → 技术方收到** | 打通真实的提交-评审流程，替代当前 Mock |
| P0 | **方案人机分工标注** | AI 生成方案时主动插入人工确认节点（改 prompt 工程） |
| P1 | **引导式输入** | AI 判断需求清晰度，模糊需求进入追问模式 |
| P1 | **通知集成** | 业务方提交后通过飞书/钉钉通知技术方 |
| P1 | **方案版本管理** | 支持查看修改历史、对比不同版本 |

### Phase 2：打通执行闭环（2-4 周）

| 优先级 | 事项 | 目标 |
|--------|------|------|
| P0 | **混合任务类型** | Workflow 节点可标记为 Agentic 节点，共享执行上下文 |
| P0 | **Agent 运行时对接** | FlowAgent 做编排层，执行下沉到 Dify/LangGraph |
| P0 | **管控后台真实数据** | 从 Agent 运行时拉取真实的任务状态和执行日志 |
| P1 | **人工确认节点运行时** | Agent 执行到人工确认节点时暂停 → 通知 → 人确认 → 继续 |
| P1 | **异常告警 + 归因** | 异常时自动告警，技术方在管控后台标注归因（搭建质量/链路设计/环境问题） |
| P1 | **历史方案参考** | AI 翻译时检索已确认方案库，参考方案骨架和关键决策 |
| P2 | **执行数据看板** | 基于真实数据的成功率、耗时、异常率统计 |

### Phase 3：规模化与生态（1-3 月）

| 优先级 | 事项 | 目标 |
|--------|------|------|
| P1 | **技能市场** | 从 Mock 升级为真实的 Skill 注册/发现/调用平台（含标准化 API schema） |
| P1 | **场景模板库** | 技术方确认的方案自动沉淀为可复用模板 |
| P1 | **SOP 文档提取** | 上传 SOP 文档（Word/PDF/飞书），AI 自动提取 Workflow 草稿 |
| P2 | **管控后台主动检测** | 方案自洽性检查、执行异常模式识别 |
| P2 | **多租户** | 支持不同团队/部门独立使用，数据隔离 |
| P2 | **权限管理** | RBAC 权限模型，细粒度控制不同角色的操作范围 |
| P3 | **审计日志** | 完整的操作审计记录 |
| P3 | **数据驾驶舱** | 业务目标达成率、人工介入率、异常率趋势等高级分析 |

### 技术债务清理

| 事项 | 说明 |
|------|------|
| 编辑器 ref 逻辑重构 | 当前用多个 `useRef` 做初始化守卫，跨场景导航可能失效，需改为 key-based 重挂载 |
| Store 持久化策略优化 | `partialize` 遗漏部分状态，`loadGeneratedFlow` 会覆盖 project.status |
| 响应式适配 | 首页特性标签在窄屏溢出，编辑器在小屏幕下布局需优化 |
| 错误边界 | API 调用失败、JSON 解析失败等场景缺少用户友好的错误提示 |

## 十一、参考研究

本文档中的设计原则参考了以下学术研究：

| 论文 | 关键发现 | 对 FlowAgent 的指导 |
|------|---------|-------------------|
