# DeskClaw Enterprise Demo 评审意见

> 评审对象：https://deskclaw-enterprise.surge.sh
> 评审日期：2026-04-02

---

## 结论

**这个 Demo 不是产品设计的交付物，是一组没有逻辑关联的静态页面。**

它展示了"我们可能需要哪些模块"，但作为产品经理的交付物，它缺失了所有应该有的东西：没有用户故事、没有核心流程、没有交互规格、没有数据流定义、没有状态机、没有异常处理。10 个模块之间没有任何串联逻辑，每个模块内部也没有完整的交互链路。

以下逐项说明。

---

## 一、最严重的问题：核心功能是空的

### 1.1 业务翻译——产品的核心价值，只有一个输入框

自定义场景页面是整个产品最核心的功能入口。当前状态：

- 一个空白输入框 + 一个"AI 分析"按钮
- 左侧列了 7 步流程，**只有第 1 步有界面，其余 6 步全部空白**
- 用户点"AI 分析"之后会发生什么？生成什么？展示在哪？怎么确认？怎么修改？**全部没有设计**

3 月 31 日会议的核心结论是"不要指望业务直接给 AI 讲明白"，需要引导式输入、模块化确认。这个 Demo 做的恰恰是让用户对着空白输入框自己讲。

**这不是"还没做完"的问题，是根本没有思考过这个流程应该怎么走。**

每一步需要定义的内容：

| 步骤 | 输入是什么 | 输出是什么 | 用户怎么操作 |
|------|-----------|-----------|-------------|
| 场景描述 | 用户自然语言 | 结构化场景摘要 | 对话式引导 |
| 类型判断 | 场景摘要 | Workflow / Agentic | AI 判断 + 用户确认 |
| 信息采集 | 类型判断结果 | 补充的业务细节 | 按类型不同的问答模板 |
| AI Plan | 完整业务信息 | 流程图草稿 / 任务配置草稿 | AI 生成 |
| 共创确认 | 草稿 | 确认后的最终方案 | 逐节点/逐模块确认 |
| 通知配置 | 确认方案 | 通知规则 | 表单 |
| 输出交付 | 全部 | 可执行配置 | 预览 + 导出 |

以上每一步都需要有界面原型和交互说明。目前 0/7。

### 1.2 Workflow 和 Agentic 完全没有区分

会议最重要的架构决策是将任务分为两大类：
- **Workflow**：确定性流程，按步骤执行，确认方式是流程图
- **Agent Tech**：目标导向，Agent 自主规划，确认方式是目标+约束+技能+评估器

这两种架构的配置方式、确认方式、监控方式、异常处理完全不同。

**Demo 的现状**：所有场景详情页结构完全一样。"财务报销"（Workflow）和"账号运营"（Agentic）的页面没有任何区别。这意味着会议讨论了两个小时的核心架构差异，在产品设计中完全没有体现。

| 维度 | Workflow 应该是什么样 | Agentic 应该是什么样 | Demo 里是什么样 |
|------|---------------------|---------------------|----------------|
| 右侧面板 | 流程图画布 | 任务配置面板（目标/技能/评估器） | 都是静态卡片 |
| 确认方式 | 逐节点确认 | 逐模块确认 | 没有确认流程 |
| 异常处理 | 节点级回溯/报警 | 重新规划/人工接管 | 没有异常处理 |

### 1.3 33 个场景，0 个能跑通

场景库有 33 个预设场景。每个场景的"开始使用"按钮点击后没有任何后续。

这意味着产品经理花时间写了 33 个场景的文案描述，但没有为任何一个场景设计完整的使用流程。**一个能跑通的场景比 33 个静态卡片有价值得多。**

### 1.4 人机协作——会议反复强调，Demo 完全缺失

会议原话："人机协作过程要避免黑盒化"、"通过钉钉或飞书通知，用户点击链接可查看任务全过程"。

Demo 中缺失的：
- "待确认"任务的确认界面——**人机协作的核心交互，没有设计**
- 任务详情页——任务只是表格里的一行，没有详情
- 异常处理流程——异常任务怎么重试/接管/跳过/回退？没有设计
- 通知集成——用户从钉钉/飞书点进来看到什么？没有设计

## 二、架构问题：10 个模块不应该在一个平台里

Demo 把业务翻译工具、运行时管控后台、Skill 基础设施全部塞进一个侧边栏。但它们是三个不同的产品：

| 产品 | 包含模块 | 用户 | 使用频率 |
|------|---------|------|---------|
| 业务翻译工具 | 自定义场景、场景库、知识中心 | 业务专家 + 技术人员 | 低频高强度 |
| 管控后台 | Agent 团队、任务监控、审计、驾驶舱 | 运维、管理者 | 高频低强度 |
| Skill 平台 | 技能市场、评估器管理 | 技术开发者 | 按需 |

把它们放在一起，等于让偶尔来配置流程的业务专家和每天盯任务的运维人员看到完全一样的导航。这不是产品设计，是功能堆砌。

## 三、逐模块评审

| 模块 | 状态 | 问题 |
|------|------|------|
| 首页 | 需调整 | "录屏+对话共创"作为核心入口不合理，技术实现难度极高，不应作为 MVP 主入口 |
| 场景库 | 需精简 | 33 个静态场景没有一个有完整交互，应砍到每种类型 1 个代表 |
| 自定义场景 | **需重做** | 7 步流程只有第 1 步有界面，核心交互全部缺失 |
| Agent 团队 | 归属错误 | 应属于管控后台，且缺少 Agent 创建/配置流程 |
| 任务监控 | 归属错误 | 应属于管控后台，缺少任务详情页和确认交互 |
| 数据驾驶舱 | 延后 | 指标缺乏业务意义（"月 Token 消耗 ¥2,340"对业务方没有意义） |
| 审计中心 | 延后 | 结构合理但属于 Phase 3 |
| 知识中心 | 可复用 | 数据结构合理 |
| 技能市场 | 可复用 | Demo 中唯一设计较完整的模块 |
| 系统设置 | 延后 | RBAC 模型合理但属于 Phase 3 |

## 四、该怎么做

### 立即要做

1. **为 1 个 Workflow 场景（如财务报销）设计完整的交互流程**：从用户输入 → 业务翻译 → 流程图生成 → 逐节点确认 → 输出，每一步都要有界面和交互说明
2. **为 1 个 Agentic 场景（如小红书运营）设计完整的交互流程**：从用户输入 → 类型判断 → 目标/技能/评估器配置 → 确认 → 输出
3. **明确 Workflow 和 Agentic 的差异化设计**：右侧面板不同、确认方式不同、输出格式不同

### 短期要做

4. 任务监控的核心交互（任务详情页、人工确认流程）
5. 通知配置的基础流程

### 延后

6. 数据驾驶舱、审计中心、权限管理、录屏+对话共创

---

## 五、总结

产品经理的交付物标准是：**用户故事 → 核心流程 → 交互规格 → 数据流定义**。

这个 Demo 跳过了所有这些步骤，直接画了一组没有内在逻辑的页面。它可以作为给客户看的愿景演示，但不能作为开发依据——因为工程师看完之后，不知道任何一个功能应该怎么做。

当前最需要的不是更多模块，而是把已有的核心模块（业务翻译、类型区分、人机协作）的交互逻辑设计清楚。
